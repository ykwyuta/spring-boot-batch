# 機能開発ドキュメント体系

機能開発（方式検討・設計・実装・テスト）を進める際のドキュメント配置ルール。

## 配置場所

```
docs/<feature-name>/
├── 01_method-study.md      # 方式検討（複数案の比較、採用案の決定）
├── 02_design.md             # 設計（インターフェース、クラス構成、シーケンス等）
├── 03_test-plan.md          # テスト計画（C1網羅の観点での分岐一覧、テストケース）
└── reviews/
    ├── 01_method-study-review.md   # 方式検討レビュー記録
    ├── 02_design-review.md         # 設計レビュー記録
    ├── 03_test-plan-review.md      # テスト計画レビュー記録
    └── 04_implementation-review.md # 実装レビュー記録
```

- `<feature-name>` は機能を表す英語のケバブケース（例: `spring-bash-service`）。
- 実装コード自体は通常のソースディレクトリ（`src/` 等）に置き、上記ディレクトリにはドキュメントのみを置く。

## 各ドキュメントの記載内容

| ファイル | 記載内容 |
| :-- | :-- |
| `01_method-study.md` | 背景・課題、検討した方式の選択肢、各選択肢の比較（メリット/デメリット）、採用方式と理由 |
| `02_design.md` | 採用方式に基づく詳細設計。クラス/モジュール構成、インターフェース、エラーハンドリング方針、設定項目 |
| `03_test-plan.md` | C1（分岐網羅）の観点での条件分岐一覧と、それぞれに対応するテストケース。期待結果を明記 |
| `reviews/*.md` | レビュー実施日、レビュー対象、指摘事項、対応方針・結果 |

## レビュー記録の書式

```
日時: 2026-06-27 12:00:00
対象ドキュメント: docs/spring-bash-service/01_method-study.md
レビュー観点: 方式の網羅性・採用理由の妥当性
指摘事項:
  - 指摘1
  - 指摘2
対応: 指摘1は反映済み。指摘2は対応不要と判断（理由: ...）
結論: 承認 / 要修正
```

## 運用ルール

- 工程（方式検討→設計→実装→テスト）を進める前に、直前の工程のレビューで「承認」が出ていることを確認する。
- ドキュメントは段階的に書き出し、セクション単位でコミットする（CLAUDE.md の常時適用ルール参照）。

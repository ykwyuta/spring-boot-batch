指摘事項:
  - 【要修正・重要】`server.address=127.0.0.1` はアプリケーション全体（HTTPコネクタ）に適用される
    設定であり、`/internal/tasks/execute` だけでなく、本プロジェクトが既に依存している
    `spring-boot-starter-thymeleaf` 等で提供される既存・将来の業務用画面・APIもすべて
    localhost限定になってしまう。本機能のためにアプリ全体の到達性を変更することの是非
    （業務用エンドポイントを外部公開する必要がある場合に矛盾しないか）が「9. 既存コードへの
    影響範囲」に記載されておらず、検討漏れになっている。最低限、以下のいずれかの対応を
    設計に追記すべき。
      (a) 現状デモ用途でアプリ全体を外部公開する予定がないことを確認し、その前提を明記する。
      (b) 業務用エンドポイントを将来外部公開する想定がある場合は、`server.address` 全体限定では
          なく、Spring MVCのインターセプタ等で `/internal/**` のみリクエスト元IPを
          `127.0.0.1`/`::1` に制限する方式（または別ポートの management server 的な構成）に
          変更する。

  - 【軽微】`TaskExecutionException` に `errorCode`（`TASK_NOT_FOUND` か `TASK_EXECUTION_FAILED` か）

結論: 要修正

再修正のポイント:
  - 「7. 設定項目」または「9. 既存コードへの影響範囲」に、`server.address=127.0.0.1` が
    アプリ全体（既存・将来の業務エンドポイントを含む）に影響する点を明記し、その影響が
    現状の要件（デモ用途・外部公開の予定なし等）と矛盾しないことを確認した上でその旨を記載する。
    矛盾する可能性がある場合は、`/internal/**` のみを対象にしたIPアドレス制限
    （`HandlerInterceptor` 等によるパス単位の制御）への変更を検討し設計に反映する。
  - 上記対応により、方式検討レビューで指摘された「バインドアドレス・認可の有無とリスク明記」が
    既存システムへの副作用まで含めて妥当であることを示せれば、再レビューで承認可能と判断する。
  - 422のレスポンス組み立て方法、例外とerrorCodeの対応方針について一言補足があると、
    実装フェーズでの解釈のブレを防げる（必須ではないが推奨）。

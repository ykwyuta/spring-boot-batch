# 方式検討: 起動済み Spring Boot プロセスの Service を bash から実行する

## 背景・課題

### 要望の整理

すでに起動・稼働している Spring Boot アプリケーション（JVM プロセス）に対して、外部の bash シェルから
何らかのコマンドを実行することで、そのプロセス内で管理されている Service Bean のメソッドを呼び出したい。

### 制約条件

- bash と Spring Boot アプリケーションは**別プロセス**である。そのため、メソッド呼び出しは何らかの
  プロセス間通信（IPC）を経由する必要がある（直接 Java のメソッドを呼ぶことはできない）。
- 「起動済みの」プロセスに対して実行する、という要件から、アプリケーション起動時に何らかの
  受け口（リスナー）を常駐させておく必要がある。
- bash から簡単に呼び出せること（特別なクライアントツールの追加インストールなしで `curl` や
  標準コマンドで叩けることが望ましい）。
- 既存リソースとの整合性。本リポジトリは以下の構成。
  - ビルドツール: Gradle（`build.gradle`）
  - 依存: `spring-boot-starter-webmvc`（Spring MVC）, `spring-boot-starter-thymeleaf`,
    `mybatis-spring-boot-starter`, Lombok, PostgreSQL ドライバ
  - Spring Boot Actuator は**未導入**
  - Spring Shell, CRaSH 等の管理シェル系ライブラリは**未導入**
  - `DemoApplication.java` のみが存在し、Service・Controller クラスはまだ存在しない（ゼロから追加する）
- セキュリティ面: 外部プロセスから任意のメソッドを実行できる経路を作ることになるため、
  最低限のアクセス制御（待受インタフェースの限定、認可、入力検証）を考慮する必要がある。

以上を踏まえ、bash プロセスから稼働中の JVM 内 Service メソッドを呼び出すための方式を比較検討する。

## 検討した方式の選択肢

### 選択肢A: REST Controller を追加し、bash から `curl` で呼び出す

Spring MVC の `@RestController` でエンドポイントを実装し、その中で対象 Service のメソッドを呼び出す。
bash 側からは `curl http://localhost:8080/api/xxx` のように HTTP リクエストを送るだけで実行できる。

**メリット**
- `spring-boot-starter-webmvc` が既に依存に含まれており、追加ライブラリが不要。
- bash からの呼び出しは `curl`（多くの環境に標準搭載）のみで完結し、専用クライアントが不要。
- Spring MVC は本プロジェクトで既に採用されているアーキテクチャであり、Controller → Service という
  一般的な層構成にそのまま乗せられる。実装パターンに馴染みがあり保守しやすい。
- リクエスト/レスポンスの JSON 変換、バリデーション（`@Valid` 等）、例外ハンドリング
  （`@ExceptionHandler`）など Spring MVC の標準機能をそのまま使える。
- HTTP ベースなので、将来的に認証（Spring Security）やログ出力、レート制限などを追加しやすい。

**デメリット・リスク**
- 「管理用の実行口」を業務 API と同じ Web レイヤーに混在させると、設計上の責務が曖昧になりうる
  （ただしパスを `/internal/**` 等に分離すれば対応可能）。
- 何も対策しないとエンドポイントが誰からでも呼べてしまうため、認可・IP制限・ローカル限定バインド
  等を別途設計する必要がある。
- HTTP サーバーが必須になる（ただし本プロジェクトは Web アプリであり、既にサーバーが起動している）。

**実装コスト・既存コードへの影響**
- 低。新規に Service クラスと、それを呼び出す `@RestController` を 1 つ追加するだけ。
- 既存の `build.gradle` の変更は不要（`spring-boot-starter-webmvc` が既存依存にあるため）。

---

### 選択肢B: Spring Boot Actuator のカスタムエンドポイントを追加し、`curl` で呼び出す

`spring-boot-actuator` を依存に追加し、`@Endpoint` / `@WriteOperation` 等でカスタム管理エンドポイントを
実装する。Actuator の `/actuator/**` パス経由で `curl` から呼び出す。

**メリット**
- Actuator の管理機能（ヘルスチェック、メトリクス等）と統一的な管理エンドポイント群に統合できる。
- Actuator にはエンドポイントの有効化/無効化、公開対象の制御、管理用ポート分離（`management.server.port`）
  など、運用上のアクセス制御機構が標準で用意されている。

**デメリット・リスク**
- `spring-boot-starter-actuator` という新規依存の追加が必要（本プロジェクトには未導入）。
- カスタム `@Endpoint` の実装方法は通常の `@RestController` よりも独自の API・アノテーション
  （`@Endpoint`, `@ReadOperation`, `@WriteOperation` 等）を学習・適用する必要があり、
  単純な「Service のメソッドを1つ呼ぶだけ」の要件に対しては学習コスト・実装コストが
  選択肢Aより高い。
- 本来 Actuator は運用監視（ヘルス・メトリクス・環境情報など）のための仕組みであり、
  任意の業務処理を実行する用途への流用は責務的にやや不自然。

**実装コスト・既存コードへの影響**
- 中。`build.gradle` への依存追加、`application.properties` でのエンドポイント公開設定、
  カスタム Endpoint クラスの実装が必要。

---

### 選択肢C: JMX に MBean として Service を公開し、`jmxterm` 等から呼び出す

Service クラスに `@ManagedResource` / `@ManagedOperation`（Spring の JMX 連携）を付与し、JVM の
JMX 経由でメソッドを公開する。bash からは `jmxterm` や `jconsole` のコマンドラインモード、
あるいは `javax.management` を使った自作クライアントで呼び出す。

**メリット**
- JVM 標準機能であり、追加の HTTP エンドポイントを公開しなくて済む（ネットワークサーフェスを増やさない）。
- Spring の `@ManagedOperation` 等を使えば実装自体は比較的簡潔。

**デメリット・リスク**
- bash から直接 JMX を叩く標準コマンドは存在せず、`jmxterm`（別途ダウンロード/インストールが必要な
  サードパーティツール）等を介する必要がある。「bash から簡単に」という要件に対して
  ハードルが高い。
- JMX のリモート接続を有効化するには `-Dcom.sun.management.jmxremote` 等の JVM 起動オプション設定が
  別途必要で、アプリケーション起動方法（起動スクリプト）にも変更が及ぶ。
- 認証・SSL 設定など JMX 自体のセキュリティ設定が煩雑。
- 本プロジェクトに JMX 関連の実装・設定が一切ないため、ゼロから JMX 連携の知識・設定を
  持ち込むことになり、学習コスト・運用コストが高い。

**実装コスト・既存コードへの影響**
- 中〜高。Service への注釈付与、JVM 起動オプションの変更、bash 側クライアントツールの導入が必要。

---

### 選択肢D: Spring Shell（remote）や CRaSH シェルを導入する

`spring-shell` や CRaSH のような、JVM 内に対話型シェルを立ち上げ、SSH や Telnet 経由で接続して
コマンドを実行できるライブラリを導入する。

**メリット**
- 対話的に複数コマンドを実行したい場合に強力（コマンド体系・補完・ヘルプなどが提供される）。

**デメリット・リスク**
- 本プロジェクトに導入実績がなく、新規依存の追加と学習コストが大きい。
- bash から非対話的に1コマンドだけ実行したいという今回の要件に対しては機能過剰（オーバーエンジニアリング）。
- SSH/Telnet 接続の口を新たに開けることになり、認証・鍵管理等のセキュリティ設計負荷が増える。
- CRaSH は長期的なメンテナンス状況が不安定で、最新の Spring Boot バージョン（本プロジェクトは
  Spring Boot 4.0.7 系）との組み合わせでの動作実績調査が必要になる。

**実装コスト・既存コードへの影響**
- 高。新規依存追加、シェルコマンドクラスの実装、起動設定、接続クライアントの用意が必要。

---

### 選択肢E: Unix ドメインソケット／名前付きパイプによる自作プロトコル

アプリケーション起動時に Unix ドメインソケット（または FIFO）をリスンするスレッドを立ち上げ、
bash からは `socat` や `nc` 等でソケットに書き込み、自作の簡易プロトコルでコマンドを送信する。

**メリット**
- ネットワークポートを開けずに済む（同一ホスト内通信に限定でき、外部からの到達性がない）。
- HTTP より低オーバーヘッド。

**デメリット・リスク**
- 自作プロトコル（コマンドの書式、レスポンス形式、エラー表現等）をゼロから設計・実装する必要があり、
  実装コスト・テストコストが最も高い。
- bash 側も `socat`/`nc` 等の追加コマンド（標準では入っていない環境もある）に依存する。
- Spring Boot の標準的な仕組み（DI、MVC、例外処理等）にそのまま乗せられず、独自にスレッド管理・
  シリアライズ処理を書く必要がある。
- 将来的な拡張（認可、ロギング、監視）も全て自前実装になる。

**実装コスト・既存コードへの影響**
- 高。既存資産（Spring MVC 等）を活用できず、もっとも実装量が大きい。

## 比較表

| 観点 | A: REST Controller | B: Actuator カスタムEndpoint | C: JMX + jmxterm | D: Spring Shell/CRaSH | E: 自作ソケットプロトコル |
| :-- | :-- | :-- | :-- | :-- | :-- |
| 追加依存ライブラリ | 不要（既存） | 必要（actuator） | 不要（JVM標準だが設定変更） | 必要（spring-shell/CRaSH） | 不要だがbash側に`socat`等が必要な場合あり |
| bash からの呼びやすさ | 高（`curl`のみ） | 高（`curl`のみ） | 低（`jmxterm`等が必要） | 中（SSH/Telnetクライアントが必要） | 中〜低（`nc`/`socat`必要、自作プロトコル） |
| 実装コスト | 低 | 中 | 中〜高 | 高 | 高 |
| 既存コードとの整合性 | 高（既存のWebMVC構成にそのまま乗る） | 中（新たなレイヤーの追加） | 低（JMX利用実績なし） | 低（シェル系ライブラリの利用実績なし） | 低（独自実装が中心） |
| セキュリティ設計のしやすさ | 高（Spring Security等を後付け可能） | 高（Actuator標準の制御機構あり） | 中（JMX自体のセキュリティ設定が必要） | 中（SSH鍵管理等が必要） | 低（全て自前実装） |
| Spring Bootとの統合の自然さ | 高 | 中（本来は運用監視用途） | 中 | 低（オーバースペック） | 低 |

## 採用方式と理由

**選択肢A「Spring Web (REST Controller) のエンドポイントとして実行対象の Service メソッドを公開し、
bash から `curl` で呼ぶ」方式を採用する。**

採用理由は以下の通り。

1. **既存資産との整合性が最も高い**: 本プロジェクトは既に `spring-boot-starter-webmvc` を
   依存に持ち、Spring MVC ベースの Web アプリケーションとして構成されている。Controller → Service
   という構成は Spring MVC の標準的な層構造そのものであり、新規ライブラリの追加なしに
   既存のアーキテクチャ・ビルド構成（`build.gradle`）にそのまま組み込める。
2. **実装コストが最小**: Actuator（選択肢B）や Spring Shell/CRaSH（選択肢D）のような
   新規依存追加・新しい API の学習が不要であり、独自プロトコル実装（選択肢E）のような
   ゼロからの設計も不要。Service クラスと `@RestController` を1つ追加するだけで実現できる。
3. **bash からの呼び出しが最も簡単**: `curl` は大半の Linux/bash 環境に標準搭載されており、
   `jmxterm`（選択肢C）や `socat`/SSH クライアント（選択肢D, E）のような追加ツールの
   インストールを利用者に要求しない。要望にある「bash から実行できるようにする」という
   目的に対して最も直接的に応える。
4. **保守性・拡張性**: Spring MVC 標準のリクエスト/レスポンス処理（JSON シリアライズ、
   `@Valid` によるバリデーション、`@ExceptionHandler` による例外処理）をそのまま活用できるため、
   将来的な機能追加（パラメータの拡張、認可制御の追加等）も Spring の標準機能の範囲内で
   対応できる。
5. **セキュリティ対策の見通しが立てやすい**: HTTP ベースであるため、将来的に Spring Security
   による認証・認可、IP アドレス制限、内部用パス（例: `/internal/**`）への分離といった
   標準的な対策を段階的に追加しやすい。本ドキュメントの設計フェーズ（`02_design.md`）で
   具体的なセキュリティ方針（バインドアドレスの制限、エンドポイントの公開範囲、必要に応じた
   簡易認証等）を定める。

選択肢B（Actuator）は運用監視目的のエンドポイント基盤であり、業務処理の実行という今回の用途には
責務的にやや不自然である上、新規依存の追加と Actuator 独自 API の学習コストが発生するため
見送る。選択肢C（JMX）・D（Spring Shell/CRaSH）・E（自作ソケット）は、いずれも
「bash から簡単に呼び出せる」という要件に対して追加ツールの導入や独自実装が必要になり、
実装コスト・運用コストの両面で選択肢Aに劣るため採用しない。

以上より、Service の実行用エンドポイントを `@RestController` として実装し、
bash からは `curl -X POST http://localhost:<port>/internal/<path>` の形式で呼び出す方式を
次の設計フェーズ（`02_design.md`）の前提として採用する。
